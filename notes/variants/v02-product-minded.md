## 产品开发流程：每个项目的标准动作

每个项目都走这个闭环，不是走形式，是养成产品直觉：

```
用户调研（问 3 个同学）→ 1 页 PRD → UI 设计 → 开发 → 部署上线 → 埋点 → 数据复盘
```

### 1 页 PRD 模板

```markdown
## [产品名] PRD
- 目标用户：谁？（具体到人，不要写"所有开发者"）
- 核心痛点：他们现在怎么解决这个问题？为什么不够好？
- 解决方案：一句话描述你要做什么
- 成功指标：DAU / 留存率 / 完成率（选一个核心指标）
- 认知可供性检查：用户第一眼能看懂怎么用吗？
- MVP 范围：砍到不能再砍，只留核心功能
- 技术选型理由：为什么用这个技术栈？（不是"因为流行"，是"因为..."）
```

### 用户调研（轻量版）

不需要正式访谈。每个项目开始前，找 3 个目标用户（同学、朋友），问 3 个问题：
1. 你现在怎么解决这个问题？
2. 最烦的是什么？
3. 如果有个工具能 ___，你会用吗？

把回答记下来，写进 PRD。这 5 分钟的调研会让你的产品方向完全不同。

## 做产品：12 个项目，12 篇文章

### Phase 1：跑通闭环（Week 1-2）

目标：跑通"调研 → PRD → 开发 → 上线 → 看数据"的完整产品闭环。

**P1. 个人网站 v0.1** — 你的线上产品主页
- 用户调研：问 3 个同学"你会怎么了解一个候选人？"
- PRD 核心：面试官 30 秒内能看到你是谁、做过什么、怎么联系
- 认知可供性：导航一目了然，项目卡片可点击，CTA 明确
- 做法：`bunx create-next-app`，Cursor 写，Vercel 部署，Umami 埋点
- 成功指标：上线后一周内有 20+ 独立访客（发到朋友圈/群里）
- 技能覆盖：C4 TypeScript 全栈、C7 AI 辅助开发
- 文章：《产品思维做个人网站：从用户调研到数据验证》

**P2. 求职追踪器** — 你和同学都会用的工具
- 用户调研：问 3 个正在找工作的同学"你怎么管理投递进度？"
- PRD 核心：比 Excel 好用，比 Notion 轻量，专注求职场景
- 做法：Next.js + SQLite，CRUD 全栈闭环，邀请 3 个同学试用
- 成功指标：3 个同学连续使用 1 周（7 日留存）
- 数据复盘：哪个功能用得最多？哪个页面跳出率高？
- 技能覆盖：C4 全栈、C2 系统设计
- 文章：《为什么我没用 Notion 而是自己做了一个求职追踪器——产品决策背后的技术选型》

**P3. AI 周报生成器** — 从工具到产品的第一步
- 用户调研：问 3 个有写周报需求的人"你每周花多久写周报？"
- PRD 核心：一条命令生成周报，但输出格式要让领导满意
- 做法：Git log → LLM API → 结构化周报，做成 CLI + Web 两个版本
- 成功指标：自己连续用 4 周 + 至少 1 个同学也在用
- 技能覆盖：C5 AI 集成、C3 技术写作
- 文章：《我的第一个 AI 产品：从"能用"到"有人用"差了什么》

### Phase 2：产品深度（Week 3-7）

目标：每个项目有真实用户，学会用数据驱动迭代。

**P4. 算法刷题伴侣** — 让刷题不再孤独
- 用户调研：问同学"刷题最大的困难是什么？"（大概率是"刷完就忘"）
- PRD 核心：不是又一个题库，是科学复习系统 + AI 提示（不给答案）
- 做法：SM-2 间隔重复 + AI 生成渐进提示，PostgreSQL + Drizzle
- 行动可供性：今天该复习哪题一目了然，不需要用户自己判断
- 成功指标：5 个用户的 7 日留存率 > 50%
- 技能覆盖：C1 算法思维、C4 全栈
- 文章：《间隔重复算法 SM-2 实现：产品设计如何影响技术选型》

**P5. 简历定制器** — 帮同学也帮自己
- 用户调研：问 3 个同学"你投不同岗位会改简历吗？怎么改？"
- PRD 核心：粘贴 JD + 上传简历 → AI 分析匹配度 → 给出修改建议
- 做法：Vercel AI SDK 流式输出，文件上传，Prompt 工程进阶
- 认知可供性：匹配度用颜色可视化（红/黄/绿），修改建议可一键应用
- 成功指标：10 个用户使用，平均每人定制 3+ 份简历
- 技能覆盖：C5 AI 集成、C4 全栈
- 文章：《Prompt 工程实战：让 AI 理解 JD 的产品设计思路》

**P6. 本地模型竞技场** — 有流量潜力的内容产品
- 用户调研：搜"M5 MacBook 跑模型"，看社区里大家在问什么
- PRD 核心：不只是跑分工具，是"帮你选模型"的决策产品
- 做法：同一 Prompt 发给多个本地模型，对比速度和质量，给出推荐
- 行动可供性：用户选择使用场景（代码/翻译/对话），系统推荐最佳模型
- 成功指标：文章阅读量 1000+，工具页面 UV 100+
- 技能覆盖：C8 本地 AI、C3 技术写作
- 文章：《M5 MacBook Pro 本地 LLM 选型指南：不只是跑分，是产品决策》

**P7. 一键部署 CLI** — 把运维经验产品化
- 用户调研：问 3 个同学"你部署项目最烦的是什么？"
- PRD 核心：交互式选择技术栈 → 自动生成 Dockerfile + GitHub Actions + 部署
- 做法：Commander.js CLI + 模板引擎，支持 Next.js / Node / Python 项目
- 成功指标：GitHub Star 20+，至少 5 个人用它部署过项目
- 技能覆盖：C6 DevOps、C4 全栈
- 文章：《把运维经验变成产品：一个部署 CLI 的产品设计与技术实现》

**P8. 文档问答机器人** — RAG 的产品化
- 用户调研：问同学"你读技术文档最痛苦的是什么？"
- PRD 核心：不是通用 ChatBot，是"课程文档助手"——上传课件就能问答
- 做法：上传 → 分块 → Embedding → 向量检索 → LLM 回答
- 认知可供性：回答附带原文引用，用户能验证准确性
- 成功指标：5 个同学用它复习，平均每人问 10+ 个问题
- 数据复盘：哪些问题回答得好？哪些检索失败？精度如何提升？
- 技能覆盖：C5 AI 集成、C2 系统设计
- 文章：《从零实现 RAG 产品：检索精度从 60% 到 85% 的迭代过程》

### Phase 3：综合展示 + 面试（Week 8-12）

目标：整合作品，用产品数据讲面试故事。

**P9. 个人网站 v2.0** — AI 原生作品集 + 产品数据仪表盘
- PRD 核心：面试官能看到每个项目的用户数据（DAU、留存、反馈）
- 做法：在 P1 基础上加：项目展示页（带数据）、技术博客、AI 聊天助手
- 认知可供性：项目卡片展示核心指标，点进去有完整产品故事
- 成功指标：面试官平均停留时间 > 3 分钟
- 技能覆盖：C2 系统设计、C4 全栈
- 文章：《个人网站从名片到产品：用数据讲述你的项目故事》

**P10. AI 运维助手** — 运维经验的 AI 升级
- PRD 核心：日志分析 + 异常检测 + 告警摘要，面向运维团队
- 做法：接入日志流 → AI 分析异常 → 自动生成告警摘要
- 成功指标：在自己的项目上跑 2 周，检测到 5+ 个真实异常
- 技能覆盖：C6 DevOps、C5 AI 集成
- 文章：《AI + DevOps：用 LLM 做智能日志分析的产品化思路》

**P11. GitHub Code Review Bot** — 开发者工具产品
- PRD 核心：不是通用 review，是针对特定规范（你团队的代码规范）的定制 review
- 做法：GitHub Webhook → 读取 PR diff → AI 审查 → 评论回写
- 成功指标：在自己的 12 个项目上启用，处理 20+ 个 PR
- 技能覆盖：C5 AI 集成、C9 开源贡献
- 文章：《构建 AI Code Review Bot：从产品需求到 GitHub Marketplace》

**P12. 开源贡献** — 在别人的产品里证明自己
- 做法：给你用过的工具（shadcn/ui、Drizzle、Ollama 等）提 PR
- 产品思维：找的不是随便一个 bug，是影响用户体验的问题
- 成功指标：PR 被合并
- 技能覆盖：C9 开源贡献、C3 技术写作
- 文章：《我的第一个开源贡献：从用户视角发现问题到 PR 被合并》

## 技术文章：产品决策背后的技术选型

每篇文章的核心不是"我用了什么技术"，而是"我为什么做这个决策"。

### 文章公式

```
文章 = 用户痛点 + 产品决策 + 技术选型理由 + 数据验证
```

不要写《我用 Next.js 做了一个网站》。
要写《为什么求职追踪器我选了 SQLite 而不是 PostgreSQL——从用户场景倒推技术选型》。

### 结构（800-2000 字）

1. **用户痛点**（2 句话）— 谁遇到了什么问题
2. **产品决策**（200 字）— 我决定怎么解决，为什么这样而不是那样
3. **技术选型**（500 字）— 为什么选这个技术栈，关键代码
4. **数据验证**（200 字）— 上线后数据怎么说，用户反馈是什么
5. **一句话总结** — 可复用的产品+技术经验

### 发在哪

- **个人网站**（P9）— 主阵地，带产品数据
- **掘金** — 选 6 篇精华发
- **Dev.to** — 选 3 篇翻译发，外企加分

## AI 辅助开发：产品经理 + 工程师的双重视角

### 工作流

```
你做用户调研 → 你写 PRD → AI 帮你设计 UI → AI 生成代码 → 你审查 → 你部署 → 你看数据 → 你写文章
   (产品)        (产品)       (C7)           (C7)        (C2)      (C6)      (产品)     (C3)
```

### 工具

| 工具 | 什么时候用 |
|------|-----------|
| Claude Code | 架构设计、多文件修改、PRD 到代码的转化 |
| Cursor | 日常写代码：补全、内联编辑、快速问答 |
| v0.dev | 前端 UI：描述组件需求，直接生成 |
| Ollama | 本地模型：P6 和 P8 的核心依赖 |
| Umami | 产品数据：每个项目的 DAU、留存、行为分析 |

### 边界

- AI 生成的每一行代码你都要能解释
- 产品决策是你做的，不是 AI 做的
- 用户调研是你做的，不是 AI 猜的
- AI 帮你写得更快，但"做什么"和"为什么做"是你的判断

## 技术栈

一套 TypeScript 打通全栈 + 产品分析工具链。

| 层 | 选择 | 为什么 |
|----|------|--------|
| 语言 | TypeScript | 类型安全，前后端统一 |
| 框架 | Next.js 15 | 全栈一体，SSR/SSG/API 都有 |
| UI | Tailwind + shadcn/ui | 快，好看，认知可供性好 |
| 数据库 | PostgreSQL (Neon) + Drizzle | Serverless 友好，TS 原生 ORM |
| AI | Vercel AI SDK + Ollama | 云端流式 + 本地推理 |
| 部署 | Vercel + Docker | 零配置 + 容器化 |
| CI/CD | GitHub Actions | 行业标准 |
| 运行时 | Bun | 快 |
| 数据分析 | Umami | 开源、自部署、隐私友好 |

---

> 核心理念：做产品 → 学技术 → 看数据 → 写文章 → 赢面试。你不是一个只会写代码的工程师，你是一个能从用户需求出发、用技术解决问题、用数据验证结果的产品工程师。3 个月后，面试官问你任何问题，你都能说"我做过，数据是这样的"。
